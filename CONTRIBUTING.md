# Contributing to MCX

Thanks for your interest in MCX! This project is still in an early design/prototype phase, but the goal is to grow it
into a serious, community-driven platform.

## How to get started

1. **Read the README** to understand the high-level vision and roadmap.
2. Check the `docs/` directory for design documents such as the Lua API draft.
3. Look at the open issues, especially those that describe design work or small, well-scoped tasks.

For now, most contributions will be:

- design discussions (issues, comments),
- documentation updates and corrections,
- small prototypes (once the implementation starts).

## Issues and discussions

- Use issues to propose changes, ask questions, or discuss design.
- Try to keep one clear topic per issue.
- When possible, reference existing docs (README, files in `docs/`) so the conversation stays grounded.

## Pull requests

For early stages of the project:

- Prefer **small, focused PRs** over large, sweeping changes.
- Explain in the PR description:
  - what you changed,
  - why you changed it,
  - how it relates to an existing issue (if applicable).
- If your change affects the design (API shape, behavior), update the relevant doc in `docs/` as part of the PR.

## Code style and languages

The exact implementation stack is still being defined (see the roadmap), but some general principles apply:

- Keep things simple and readable.
- Favor clear naming over clever tricks.
- Document public APIs and concepts.

More detailed guidelines will be added once the implementation solidifies.

## Code of Conduct

By participating in this project, you agree to uphold a respectful and welcoming environment. A formal Code of Conduct
will be added, but the short version is:

- be kind,
- assume good intent,
- disagree on ideas without attacking people.
